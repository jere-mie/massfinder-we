# Backend API — Scope & Design

This document defines the scope, architecture, database schema, and migration plan for a lightweight Go backend API that replaces the static `churches.json` / `events.json` files with live database-backed endpoints.

---

## Motivation

Currently the frontend fetches two static JSON files deployed alongside the site:

- `public/churches.json` — all church schedule data
- `public/events.json` — all parish events

These files are regenerated by the Python scraper and require a full frontend redeploy (GitHub Pages push) every time data changes. Moving to a backend API decouples the data lifecycle from the frontend deployment:

- The scraper writes directly to a SQLite database.
- The API serves live data with no frontend redeploy needed.
- Events and mass times update independently of the site.

---

## Technology Choice

### Language: Go

Go is chosen over Python for the following reasons:

| Concern | Go | Python (FastAPI + uvicorn) |
|---|---|---|
| Cold start latency | ~0 ms (compiled binary) | ~300–600 ms (interpreter + import chain) |
| Memory footprint | ~8–15 MB resident | ~80–150 MB resident |
| Deployment artifact | Single static binary, no runtime deps | Python environment + pip packages |
| Concurrency model | goroutines (OS-thread-backed, cheap) | asyncio (single-threaded event loop) |
| SQLite integration | `modernc.org/sqlite` (pure Go, no CGO) | `sqlite3` stdlib (C extension) |

The scraper **stays in Python** — it already works and language consistency between the scraper and API is not required since they communicate through a shared SQLite file, not over HTTP.

### Libraries

| Concern | Library | Rationale |
|---|---|---|
| HTTP routing | [`go-chi/chi`](https://github.com/go-chi/chi) | Lightweight, idiomatic, zero external deps beyond stdlib |
| SQLite driver | [`modernc.org/sqlite`](https://pkg.go.dev/modernc.org/sqlite) | Pure Go, no CGO, cross-compiles to Linux without toolchain hassle |
| CORS middleware | [`rs/cors`](https://github.com/rs/cors) | Well-audited, chi-compatible, minimal config |
| Config | `os.Getenv` (stdlib) | No config library needed at this scale |

No ORM. Direct `database/sql` queries are explicitly preferred: they are faster, easier to reason about, and the query set is small and read-only.

## Database Schema

The JSON arrays in `churches.json` normalize into five tables. The scraper writes to this schema directly using Python's `sqlite3` stdlib module — no new Python dependencies required.

```sql
-- Core church identity and contact info
CREATE TABLE churches (
    id                 TEXT PRIMARY KEY,        -- e.g. "st-john-the-baptist-amherstburg"
    name               TEXT NOT NULL,
    family_of_parishes TEXT,
    address            TEXT NOT NULL,
    lat                REAL NOT NULL,
    lng                REAL NOT NULL,
    map_url            TEXT,
    website            TEXT,
    bulletin_website   TEXT,
    phone              TEXT                     -- "+15197365418" format
);

-- Weekend and daily Mass times (type distinguishes the two)
CREATE TABLE masses (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    church_id  TEXT NOT NULL REFERENCES churches(id) ON DELETE CASCADE,
    type       TEXT NOT NULL CHECK(type IN ('regular', 'daily')),
    day        TEXT NOT NULL,                   -- "Monday" … "Sunday"
    time       TEXT NOT NULL,                   -- HHMM 24-hour, e.g. "1830"
    note       TEXT
);

-- Confession and adoration windows (both are a day + start + end range)
CREATE TABLE time_ranges (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    church_id  TEXT NOT NULL REFERENCES churches(id) ON DELETE CASCADE,
    type       TEXT NOT NULL CHECK(type IN ('confession', 'adoration')),
    day        TEXT NOT NULL,
    start_time TEXT NOT NULL,                   -- HHMM
    end_time   TEXT NOT NULL,                   -- HHMM
    note       TEXT
);

-- Parish events extracted from bulletins
CREATE TABLE events (
    id                   TEXT PRIMARY KEY,      -- 8-char base36 e.g. "k7m2x9p1"
    title                TEXT NOT NULL,
    description          TEXT,
    church_id            TEXT,                  -- nullable: may be family-level
    church_name          TEXT,
    family_of_parishes   TEXT,
    date                 TEXT NOT NULL,         -- YYYY-MM-DD
    start_time           TEXT,                  -- HHMM, nullable
    end_time             TEXT,                  -- HHMM, nullable
    location             TEXT,
    source_bulletin_link TEXT,
    source_bulletin_date TEXT,                  -- YYYY-MM-DD
    extracted_at         TEXT                   -- ISO 8601 timestamp
);

-- Event tags (normalized from events[].tags array)
CREATE TABLE event_tags (
    event_id TEXT NOT NULL REFERENCES events(id) ON DELETE CASCADE,
    tag      TEXT NOT NULL,
    PRIMARY KEY (event_id, tag)
);

-- Recommended indexes
CREATE INDEX idx_masses_church     ON masses(church_id);
CREATE INDEX idx_masses_day        ON masses(day);
CREATE INDEX idx_time_ranges_church ON time_ranges(church_id);
CREATE INDEX idx_events_date       ON events(date);
CREATE INDEX idx_event_tags_event  ON event_tags(event_id);
```

## API Endpoints

All endpoints are **read-only GET**. No write endpoints are exposed. The response shapes are designed to be drop-in replacements for the existing JSON files — the frontend hooks require only a URL change.

### `GET /churches`

Returns all churches with their full schedule data assembled from the normalized tables. Shape matches the existing `Church[]` TypeScript type.

```json
[
  {
    "id": "st-john-the-baptist-amherstburg",
    "name": "St. John the Baptist",
    "familyOfParishes": "Amherstburg-Harrow Catholic Family of Parishes",
    "address": "225 Brock St, Amherstburg, ON N9V 2H3",
    "coordinates": [42.1031731983902, -83.10377441840853],
    "map": "https://maps.app.goo.gl/...",
    "website": "http://www.ahcfop.ca/",
    "bulletin_website": "https://www.ahcfop.ca/bulletins",
    "phone": "+15197365418",
    "masses": [{ "day": "Saturday", "time": "1700" }],
    "daily_masses": [{ "day": "Wednesday", "time": "0900" }],
    "confession": [{ "day": "Saturday", "start": "0945", "end": "1030" }],
    "adoration": [{ "day": "Wednesday", "start": "0930", "end": "2130" }]
  }
]
```

### `GET /churches/{id}`

Returns a single church by its slug ID. `404` if not found.

### `GET /events`

Returns all events sorted by date ascending. Optionally filter with query params:

| Param | Type | Description |
|---|---|---|
| `after` | `YYYY-MM-DD` | Exclude events before this date. Use `after=today` as a convenience value. |
| `family` | string | Filter by `family_of_parishes` value. |
| `tag` | string | Filter by event tag (e.g. `liturgy`, `community`). |

```json
[
  {
    "id": "r4mera2l",
    "title": "Monthly Healing Mass",
    "description": "All are invited to a Healing Mass...",
    "church_id": null,
    "church_name": null,
    "family_of_parishes": "North Central Windsor Catholic Family of Parishes",
    "date": "2025-12-05",
    "start_time": "1900",
    "end_time": null,
    "location": "St. Michael's Church",
    "tags": ["liturgy", "community"],
    "source_bulletin_link": "https://...",
    "source_bulletin_date": "2025-12-01",
    "extracted_at": "2025-12-01T21:30:48.144963"
  }
]
```

### `GET /health`

Returns `200 OK` with a plain text body `ok`. Used for uptime monitoring.

## CORS Configuration

The API is intentionally open for reading — no authentication headers or allowlist required.

```go
cors.New(cors.Options{
    AllowedOrigins: []string{"*"},
    AllowedMethods: []string{"GET", "OPTIONS"},
    AllowedHeaders: []string{"Accept", "Content-Type"},
    MaxAge:         86400, // cache preflight for 24 hours
})
```

`MaxAge: 86400` eliminates repeated `OPTIONS` preflight round-trips from the browser — each origin only pays the preflight cost once per day.

---

## HTTP Caching

Since data only changes when the scraper runs (typically daily or less frequently), response caching dramatically reduces both server load and user-perceived latency.

```
GET /churches  →  Cache-Control: public, max-age=3600    (1 hour)
GET /events    →  Cache-Control: public, max-age=900     (15 minutes)
GET /health    →  Cache-Control: no-store
```

When placed behind a CDN like Cloudflare (free tier), the first request per edge region populates the CDN cache. All subsequent requests in that region are served from the CDN edge — the origin server is not contacted at all. This is broadly safe because the data does not change in real-time.

If tighter freshness is needed (e.g. after a scraper run), the CDN cache can be purged via the Cloudflare API from the scraper after it finishes writing.

## Project Structure

This should live in a **separate repository** (`massfinder-api`), not inside this frontend repo.

```
massfinder-api/
├── main.go                  # Entry point: wires router, middleware, server
├── go.mod
├── go.sum
├── db/
│   ├── schema.sql           # Source of truth for the DB schema
│   └── db.go                # Opens connection pool, runs PRAGMA optimizations
├── handlers/
│   ├── churches.go          # GET /churches, GET /churches/{id}
│   └── events.go            # GET /events
└── data/
    └── massfinder.db        # SQLite file (written by scraper, read by API)
```

The `data/` directory is `.gitignored`. In production it lives on the server at a configurable path set via `DB_PATH` environment variable.

## Deployment

### Server

A single `go build -o massfinder-api .` produces one binary (~8 MB) with no runtime dependencies. It runs cleanly on any Linux VPS. The smallest available tier (e.g. 512 MB RAM, 1 vCPU) easily handles this workload.

### Environment Variables

| Variable | Default | Description |
|---|---|---|
| `DB_PATH` | `./data/massfinder.db` | Path to SQLite database file |
| `PORT` | `8080` | HTTP listen port |
| `ALLOWED_ORIGINS` | `*` | Comma-separated CORS origin allowlist (override to restrict) |

### Process Management

Run as a systemd service:

```ini
[Unit]
Description=MassFinder API
After=network.target

[Service]
ExecStart=/usr/local/bin/massfinder-api
Environment=DB_PATH=/var/lib/massfinder/massfinder.db
Environment=PORT=8080
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

## Scraper Migration

The scraper (`scraper/app.py`) currently writes to `public/churches.json` and `public/events.json`. Migration involves replacing the JSON write step with SQLite upserts.

### Changes to `scraper/app.py`

1. At startup, open a connection to the SQLite file at `DB_PATH`:

   ```python
   import sqlite3, os
   db = sqlite3.connect(os.getenv("DB_PATH", "../data/massfinder.db"))
   db.execute("PRAGMA journal_mode=WAL")  # allow concurrent readers
   db.execute("PRAGMA foreign_keys=ON")
   ```

2. After building the updated churches list, replace `json.dump(churches, ...)` with an upsert transaction:

   ```python
   def save_churches_to_db(db, churches):
       with db:
           for church in churches:
               db.execute("""
                   INSERT OR REPLACE INTO churches
                     (id, name, family_of_parishes, address, lat, lng,
                      map_url, website, bulletin_website, phone)
                   VALUES (?,?,?,?,?,?,?,?,?,?)
               """, (
                   church["id"], church["name"], church.get("familyOfParishes"),
                   church["address"], church["coordinates"][0], church["coordinates"][1],
                   church.get("map"), church.get("website"), church.get("bulletin_website"),
                   church.get("phone"),
               ))
               church_id = church["id"]
               # Delete and re-insert child rows for this church
               db.execute("DELETE FROM masses WHERE church_id = ?", (church_id,))
               db.execute("DELETE FROM time_ranges WHERE church_id = ?", (church_id,))
               for m in church.get("masses", []):
                   db.execute("INSERT INTO masses (church_id,type,day,time,note) VALUES (?,?,?,?,?)",
                              (church_id, "regular", m["day"], m["time"], m.get("note")))
               for m in church.get("daily_masses", []):
                   db.execute("INSERT INTO masses (church_id,type,day,time,note) VALUES (?,?,?,?,?)",
                              (church_id, "daily", m["day"], m["time"], m.get("note")))
               for c in church.get("confession", []):
                   db.execute("INSERT INTO time_ranges (church_id,type,day,start_time,end_time,note) VALUES (?,?,?,?,?,?)",
                              (church_id, "confession", c["day"], c["start"], c["end"], c.get("note")))
               for a in church.get("adoration", []):
                   db.execute("INSERT INTO time_ranges (church_id,type,day,start_time,end_time,note) VALUES (?,?,?,?,?,?)",
                              (church_id, "adoration", a["day"], a["start"], a["end"], a.get("note")))
   ```

3. For events, replace `save_events_json(...)` with:

   ```python
   def save_events_to_db(db, events):
       with db:
           for event in events:
               db.execute("""
                   INSERT OR REPLACE INTO events
                     (id, title, description, church_id, church_name,
                      family_of_parishes, date, start_time, end_time,
                      location, source_bulletin_link, source_bulletin_date, extracted_at)
                   VALUES (?,?,?,?,?,?,?,?,?,?,?,?,?)
               """, (
                   event["id"], event["title"], event.get("description"),
                   event.get("church_id"), event.get("church_name"),
                   event.get("family_of_parishes"), event["date"],
                   event.get("start_time"), event.get("end_time"),
                   event.get("location"), event.get("source_bulletin_link"),
                   event.get("source_bulletin_date"), event.get("extracted_at"),
               ))
               db.execute("DELETE FROM event_tags WHERE event_id = ?", (event["id"],))
               for tag in event.get("tags", []):
                   db.execute("INSERT OR IGNORE INTO event_tags (event_id, tag) VALUES (?,?)",
                              (event["id"], tag))
   ```

`PRAGMA journal_mode=WAL` is mandatory — it allows the Go API to read the database concurrently while the scraper writes, without locking errors.

### No New Python Dependencies

Only `sqlite3` (Python stdlib) is needed. The scraper's `requirements.txt` does not change.

## Frontend Migration

Only two files need to change. The API base URL should be defined once in `src/utils/constants.ts`, not duplicated across hooks.

### 1. Add to `src/utils/constants.ts`

```ts
export const API_BASE_URL = import.meta.env.PUBLIC_API_BASE_URL ?? 'https://api.mass.bornais.ca';
```

Set `PUBLIC_API_BASE_URL` in a `.env` file for local development pointing at a local API instance.

### 2. Update `src/hooks/useChurches.ts`

```ts
import { API_BASE_URL } from '../utils/constants';
// ...
fetch(`${API_BASE_URL}/churches`)
```

### 3. Update `src/hooks/useEvents.ts`

```ts
import { API_BASE_URL } from '../utils/constants';
// ...
fetch(`${API_BASE_URL}/events?after=today`)
```

The `after=today` param avoids returning expired events without client-side filtering — the API handles it.

## Open Questions / Future Scope

- **Auth for writes**: If a write API is ever needed (e.g. admin UI), add API key authentication via a bearer token checked in middleware. Do not add this until explicitly required.
- **Pagination**: `/events` could eventually need cursor-based pagination if the event history grows large. Not needed now.
- **CDN cache invalidation**: After each scraper run, call the Cloudflare purge API to immediately propagate fresh data. The scraper already has a `requests` dependency, so this is trivial to add.
- **Static JSON fallback**: During the transition, the frontend could try the API and fall back to the static JSON files. This simplifies rollback if the backend has issues.
